Human-Robot Interaction

  • Type: On-Campus Navigator
  • Target Audience: Freshmen, Exhange students, guests that need help getting to a place in school.
  • Objective: Give instructions to people in need on-campus to get to a destination.
  • Main Feature: A selection destinations such as lifts, lecture theatres, restaurants, and other landmarks, with verbal commands to guide the users.
  • Prototype Tools: Softbank Pepper, Choregraphe, Python, Photoshop
  • Team Members: Oliver, Ren, Chung Yan

Overview

The objective of this project is to program a robot, such as Softbank's Pepper or Nao, so that it can be used on campus to help out people's needs. After observing and interviewing some students on campus, we decided to program a robot navigator to guide students to specific location verbally.

Prototype

Oliver and I designed and programmed the prototype. We tried to incorporate all lifts and major locations, such as the library, canteens, lecture theatres, etc. From the small survey sample, we decided to place our robot outside LTC and LTD, where there are less traffic. We used Choregraphe to program Pepper as well as added some custom python scripts inside to achieve our goal.

Deisgn Flow

  1. Displays start menu image on tablet with QR code for HKUST path advisor
  2. Touch Pepper's head to start
    1. Pepper says: "Where do you want to go today?"
    2. Displays menu for selecting destination category
  3. User selects category
    1. User selects lift
      1. Displays lift categories
      2. User selects a lift category
      3. Displays specific lift/lift group
      4. User selects specific lift/lift group
    2. User selects LTs
      1. Displays specific LT/LT groups
      2. User selects specific LT/LT groups
    3. User selects Restaurants
      1. Displays specifc restaurant/restaurant groups
      2. User selects specific restaurant/restaurant groups
    4. User selects Others
      1. Displays other specific locations
      2. User selects a specific location
  4. Pepper gives verbal instructions to destination while giving a pointing gesture
  5. Displays Replay/End menu
    1. If replay, go back to 4
    2. If end, Pepper says: "Have a nice day, good bye" and bows. Go back to 1

A back button is added to all the intermediate selection process incase the user makes a mistake.

Final Thoughts and Review

Our initial idea was to put a QR code to path advisor with the specific route for each destination, but apparently the mobile version of the website is different from the web version, which does not contain a nearest path search. Hence, we have to go with a more tedious method of programming and have a verbal direction for each destination.

Due to the tediousness while programming, we group some destinations together and gave the same instruction. We assumed that after following the general direction that we have gave, the user would be able to see the destination or signs that points to it. Of course, this assumption may be faulty.

If given more time and/or redo the same project, I would add the following:

  • Add a QR code to the destination location on the mobile version, and display it on the tablet.
  • Make a better animation for pointing the direction.
  • Have Pepper verbally say out the selection the user has made to give them confirmation
